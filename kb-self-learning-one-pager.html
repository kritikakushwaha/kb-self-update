The problem

When AI chat fails and a human agent resolves the ticket, nobody goes back to check what went wrong with the KB. The article stays broken until someone randomly notices. Separately, customers with sparse or no KB can't get value from AI chat at all because there's nothing for it to reference.

The model: why did AI fail?

Every human-resolved ticket is a signal. The system classifies what kind, then acts accordingly.

ClassificationWhat happenedSystem response
Outdated articleArticle has wrong info. Agent corrected it.Draft a correction with evidence.
Incomplete articleArticle exists but doesn't cover this topic.Draft an addition to the article.
No article existsAgent answered from scratch. Nothing in KB.Draft a new article from the resolution.
Too complexRight info exists but user couldn't follow it.Flag for simplification — content design, not factual fix.
Action requiredAgent did something (reset account, provisioned tool). Not a knowledge gap.Propose a routing article: "This needs a human — AI creates ticket + sets expectations."
Search failureArticle exists but AI didn't find it.Retrieval/RAG problem, not content. Log separately.

For informational resolutions, the system compares the agent's answer to the KB article (or extracts knowledge if no article exists). For action-based resolutions, it proposes a different kind of article — one that helps AI route correctly next time.

Two customer journeys, same pipeline

Has existing KB

Articles go stale. Corrections needed. "Your HSA article is wrong — here's the fix, based on 3 agents who corrected it."

Building KB from scratch

No articles yet. "Your agents answered 8 GitHub 2FA questions this month. Here's a draft article from their answers."

What this can't see

Echo problem: Agent trusts the wrong article, repeats the wrong answer. Zero delta. System thinks everything's fine. Especially dangerous for health benefits, compliance, financial data.
Silent deflection: AI gives wrong answer, user accepts, no ticket created. Deflection rate looks healthy while the answer is wrong.

These are structural limitations of any system that relies on human-resolved tickets as signal. Mitigations exist (agent feedback channel, time-based review, AI conversation logging) but are separate workstreams.

What we need to align on

1. Are we logging which KB article gets surfaced per ticket? This is the prerequisite. If not, it's the first thing to build — one database column.
2. EA scope: Do we ship visibility only (article failed N times, here are the conversations — no LLM) or include the classification + diagnosis layer (LLM tells admin what's wrong)?
3. Intervention timing: Immediate (prompt the agent right after resolution: "want to save this as an article?") or retroactive (batch analysis surfaced on KB page) or both?
4. How does the system classify resolution type? LLM auto-classifies from transcript (accurate, costs per ticket) vs structured agent input at resolution (adds friction) vs hybrid?
5. Does the creation path ship alongside correction in EA? Different LLM prompt, same infrastructure. Potentially more valuable for customers without a KB.
