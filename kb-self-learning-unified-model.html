KB self-learning: unified model

A merged model combining both perspectives. Items marked with Kritika Derek New show where each idea originated. Everything here is a starting point for filtering together — not a final spec.

The problem we're solving

Right now, when AI chat uses a KB article and it doesn't help — the user submits a ticket, an agent resolves it, and that's it. Nobody goes back to check whether the article was wrong. Nobody connects the dots between "this article was shown to the user" and "a human still had to step in." The KB stays broken until someone randomly notices.

Separately, many of our customers don't have a rich KB at all. They can't get value from AI chat because there's nothing for it to reference. For them, the problem isn't fixing wrong articles — it's creating articles in the first place.

This model addresses both: maintaining existing KBs and building new ones, using the same underlying pipeline.

The model at a glance

Human resolves ticket
→
Classify what happened
→
Diagnose the KB gap
→
Surface insight + draft

Every human-resolved ticket is a signal. The system classifies what kind of signal, runs a diagnosis, and surfaces the insight to the right person. Sometimes that's a draft article update. Sometimes it's a new article proposal. Sometimes it's just "heads up, this article is failing and here are the conversations."

Step 1: why did AI fail?

Before analyzing what to fix, classify why the query reached a human. This determines what the system does next.

Incomplete article Article exists but doesn't cover the specific topic the user asked about. Missing info finding — propose addition to existing article. Kritika
No article exists No relevant article in the KB at all. Agent provided the answer from scratch. Creation path — extract knowledge, propose new article. Derek Kritika

How does the system classify these? Options: LLM classifies automatically from the transcript (higher cost, more accurate), structured agent input at resolution (adds friction), or hybrid — system guesses, agent confirms with one tap. Derek proposed a Verification step where AI re-scans the KB to distinguish search failures from true gaps. How expensive is that?
The "action required" insight: When an agent provisions a tool or resets an account, the system shouldn't try to write a how-to article. But it SHOULD create a routing article — so next time AI chat can say "I know what you need, and it requires a human agent. I've created a ticket — expect resolution within 4 hours." That's a deflection in the sense that the user gets an immediate, helpful answer. It just doesn't resolve the underlying request.

Step 3: diagnosis — two resolution types, two paths

Kritika Not every resolution is the same. The system should distinguish between informational resolutions and action-based resolutions because they produce different kinds of knowledge.

Informational resolution

Agent provides knowledge: explains a policy, gives a how-to, corrects a wrong fact. This is what feeds KB articles.

System action: Compare agent's answer to the KB article (correction path) or extract knowledge for a new article (creation path).

Action-based resolution

Agent does something: provisions access, resets credentials, escalates to another team, approves a request. The knowledge isn't "how to do X" — it's "this requires a human."

System action: Log as intent limitation. Propose a "routing article" so AI can set expectations and create a ticket next time instead of failing silently.

Why this matters: Without this split, the system might try to write a KB article that says "here's how to reset someone's account" from an agent's resolution. That's not useful — users can't reset their own accounts. The useful article is: "Account resets require IT support. When you ask, AI will create a ticket and IT typically resolves it within 2 hours."

Step 4: the LLM analysis

Derek Kritika For informational resolutions, the system runs a comparison between the agent's resolution and the KB article (or extracts knowledge if no article exists).

Correction path (article exists)

LLM receives: article content + ticket transcript + agent notes. Outputs structured findings:

  • Wrong info: article says X, agent corrected to Y. Example: "$3,650 → $4,150."
  • Missing info: agent provided information the article doesn't cover. Example: SpringHealth therapy benefit.
  • Each finding tagged general (all users) vs specific (this one person). Only general findings feed into KB changes.

Creation path (no article)

LLM extracts generalizable knowledge from the resolution. What question was asked, what the answer was, whether other users would ask the same thing.

Verification step Derek

Before concluding "no article exists," the system re-scans the KB to check if a relevant article was just buried by poor search ranking. This distinguishes true content gaps from search/retrieval failures.

How expensive is the verification step? For EA, we could skip it and only process tickets where an article was already linked (simple DB lookup). The verification step becomes valuable when we want to catch search failures too.

Step 5: accumulation — building confidence across tickets

Kritika One ticket finding something wrong is a signal. Multiple tickets finding the same thing is a pattern. The system groups findings by article + subject across tickets.

The key principle: every finding is visible to the admin from the first ticket. The question is only whether the system does the extra work of generating a draft automatically or waits for more signal.

Agent conflict detection Kritika

If two tickets correct the same fact but agents give different answers (one says "3 days" and another says "2 days"), the system surfaces a conflict alert instead of a draft. This is higher priority than a normal finding — users are actively getting inconsistent answers.

Conflict example: PTO advance notice. Jordan Blake tells one user "3 business days." Lisa Park tells another "2 business days." The article says "5 business days." All three are different. The system cannot pick one — it needs human verification before any KB update.

Prioritization

Not all findings are equal. A wrong number in an article viewed 400 times/month matters more than a minor phrasing issue in an article viewed 10 times. Findings should be ranked by signal strength — number of corroborating tickets, article view count, deflection rate.

When does this run?

Kritika Two intervention points, not one:

Immediate: on that ticket

As soon as the agent resolves the ticket, the system can proactively surface an insight right there: "It looks like you referenced info that isn't in the KB — want to save this as an article?" or "The article you used may be outdated — here's what we noticed."

This catches agents in the moment while context is fresh.

Retroactive: on the KB page

Batch analysis across multiple tickets. The KB article page shows: "X tickets suggest this article needs changes." Findings accumulated, conflicts detected, drafts generated.

This is where patterns become visible — one ticket is a signal, three is a trend.

For EA, which intervention point comes first? Immediate (per-ticket, lower lift, less intelligent) or retroactive (batch, needs accumulation logic, more powerful)? Or is the foundation just the visibility layer — "this article was surfaced N times, escalated N times, here are the conversations" — with no LLM at all?

Two customer journeys, same pipeline

Customer A: has existing KB

Their problem is maintenance. Articles go stale, policies change, new benefits get added. They need the correction path — find what's wrong and draft the fix.

Primary value: "Your Benefits article is wrong about HSA limits. Here's the fix, based on 3 agents who independently corrected it this month."

Customer B: building KB from scratch

Their problem is that they don't have articles at all. They can't get value from AI chat because there's nothing for it to reference. They need the creation path — generate articles from what agents are already telling users.

Primary value: "Your agents answered 8 GitHub 2FA questions this month. Here's a draft article based on their answers. Publish it and AI can handle these next time."

Same pipeline, different output. Correction path: article + transcript → find what's wrong → draft the fix. Creation path: transcript (no article) → extract knowledge → draft new article. The infrastructure is shared. The LLM prompt is slightly different. Both produce a draft for admin review.

New There's also a complementary approach for Customer B: allowlisted domain search at query time. Before a ticket even reaches an agent, check approved external sources (e.g., Slate public help docs). If found, deflect AND auto-generate a KB article. KB builds itself from real demand. This is upstream of the resolution drafts pipeline — it reduces the tickets that ever reach a human. Shared infrastructure, different trigger.

What this system cannot see

Kritika These are structural limitations, not implementation gaps. They must be communicated honestly — to the team and to the admin user.

Blind spot 1: the echo problem

Agent reads the KB article, trusts it, tells the user the same wrong answer. Zero delta between article and resolution. The system thinks everything is fine. The KB stays wrong.

This is the most dangerous case — especially for high-stakes content like health benefits, compliance data, financial info. A wrong HSA limit has real consequences.

Mitigation paths: Agent feedback channel (flag doubt), time-based decay (annually-changing data gets auto-flagged for review), external source monitoring.

Blind spot 2: silent deflection

AI gives a wrong answer from a wrong article. User accepts it. No ticket created. Deflection rate looks healthy while the answer is wrong. The user gets burned later (PTO request rejected, wrong HSA contribution).

Mitigation path: Log all AI conversations (even when no ticket is created). When an article is later corrected, retroactively identify which AI deflections gave wrong answers. Potentially reach out to affected users.

What the admin should see on the article page: "This analysis covers tickets where a human agent intervened. If AI resolved a query using this article and the user accepted, the system has no way to verify whether that answer was correct. X conversations were resolved by AI using this article — those are not checked."

The agent's role in this system

Agents are both the source of truth (their resolutions contain the correct answers) and a potential point of failure (they can echo wrong KB content or disagree with each other).

Agent feedback channel Kritika

The echo problem makes this essential. Agents need a way to signal "I used this article but I'm not confident it's right" — even when they followed the article's guidance.

This is broader than a thumbs button. It's about the agent having a space they understand they can use to provide feedback to the system about knowledge quality. The UI form is an open design question:

OptionSignal qualityFriction
Dedicated prompt after resolutionHigh — explicit, structuredMedium — extra step
Annotation on KB articleHighest — tied to articleHigh — new interaction
Internal notes with better affordancesVariableLow — already exists
Inline flag while composing replyHigh — in the momentLow — one click

Constraint: Whatever form it takes, the output needs to be structured enough for the system to act on — which article, what's wrong, general or specific.

Agent-side copilot Kritika

While the agent is working on a ticket, AI can proactively suggest relevant KB articles. This creates a natural moment for the agent to notice "this article is wrong" while they're actively using it. The green box: if response requires an internal KB article, AI proactively surfaces it.

Engineering considerations

Data prerequisites

  • Log KB article per ticket — which article AI surfaced. If not logging this today, it's the #1 prerequisite.
  • Log deflection outcome — did the user self-serve or did it become a human ticket?
  • Findings store — structured output per ticket from the LLM analysis.
  • KB-queue relationship — currently 1:1, will evolve. Build article lookup as a swappable function.

Cost estimate

Per resolved ticket: one LLM call (~2K tokens in, ~500 out) = $0.01–0.03. Draft generation when findings accumulate: ~$0.03 per draft. For 50 human-handled tickets/week: ~$1–3/week total LLM cost.

Batch vs real-time

Nightly batch for EA — collect the day's resolved tickets, process together. Simpler infrastructure, easier to monitor. Real-time processing is a future optimization.

Derek proposed a "Post-Mortem Workflow" that triggers automatically when a ticket is resolved. Is this the same as the nightly batch, or does it need to be synchronous/near-realtime? What's the engineering preference?

What's not in this model (intentionally)

  • Auto-publishing — too early. No approval history to calibrate confidence yet. North star, not EA.
  • Source sync — monitoring upstream docs for changes. Valuable but separate infrastructure.
  • Copy-paste diffs — detecting when agents edit KB content before sending. Interesting signal, unclear build cost.
  • Full gap detection — clustering all unmatched tickets to propose new articles. The creation path handles the most common case; full clustering is v2+.
